# 家长账号与绑定管理设计

## 1. 背景

当前项目已经具备家长端小程序页面、家长角色登录能力，以及后端 `parent_accounts`、`parent_student_relations` 两张基础表，但整条链路仍存在几个明显问题：

1. 管理后台没有独立的“家长管理”入口，无法集中管理家长账号、密码和绑定关系。
2. 家长账号目前更偏向登录时临时识别和补建，缺少稳定、可回溯的手机号主身份规则。
3. 学员档案里的 `guardianPhone` 已经承担了事实上的家长识别入口，但系统没有把“自动绑定”和“人工绑定”明确区分开。
4. 小程序端虽然已有家长首页和家长业务页面，但登录认知仍不够明确，后台规则也不足以支撑多家长、多孩子的稳定绑定。

本次设计聚焦于补齐“手机号主身份的家长账号体系 + 多对多绑定管理 + 后台管理入口 + 小程序同步登录规则”，让家长管理从隐式能力升级为明确、稳定、可维护的业务模块。

## 2. 设计目标

### 2.1 核心目标

1. 以手机号作为家长唯一主身份，建立稳定的家长账号与档案管理机制。
2. 支持“一个家长绑定多个学生、一个学生绑定多个家长”的多对多关系。
3. 学员 `guardianPhone` 变更时，系统自动维护自动绑定关系，无需后台手工兜底。
4. 后台支持家长列表、详情、密码管理，以及人工绑定/解绑学生。
5. 小程序家长端统一使用“手机号 + 密码”登录，复用现有家长首页和业务页面。

### 2.2 已确认的业务规则

1. 家长账号主身份为手机号。
2. 家长登录方式为“手机号 + 密码”，暂不接入短信验证码。
3. 系统在识别到新手机号时自动创建家长账号。
4. 自动创建账号的初始密码为手机号后 6 位。
5. 学员 `guardianPhone` 对应的家长关系自动建立绑定。
6. 学员 `guardianPhone` 变更时，系统自动解除旧手机号家长与该学生的自动绑定，并自动绑定到新手机号家长。
7. 后台手动调整形成的绑定关系独立于 `guardianPhone` 自动规则，后续手机号变更时仍然保留。
8. 本阶段后台不做家长账号启用/禁用。

### 2.3 非目标

1. 本阶段不接入短信验证码登录。
2. 本阶段不支持家长自主注册。
3. 本阶段不支持家长账号删除。
4. 本阶段不重做家长端首页和业务页面信息架构。
5. 本阶段不引入绑定审核流、复杂审计页或消息运营后台。

## 3. 方案结论

本次采用“手机号主身份 + 显式区分自动绑定/人工绑定”的方案。

核心思路如下：

1. 家长登录账号统一收口到手机号，`user_accounts.username` 直接保存规范化手机号。
2. `parent_accounts` 继续作为家长档案表，但收口为“一手机号一档案”。
3. `parent_student_relations` 继续表示家长和学生的多对多关系，并新增绑定来源语义字段，避免自动规则与人工操作互相覆盖。
4. 学员保存、导入、修改 `guardianPhone` 时自动同步家长账号和绑定关系。
5. 后台提供显式的家长管理入口，负责人工维护长效绑定和密码操作。

选择该方案的原因：

1. 能准确满足“自动绑定”和“人工绑定”并存的业务需求。
2. 能稳定支持家长手机号变化、学生监护人手机号变化、多孩子绑定、多家长绑定等真实场景。
3. 改动量明显小于重建整套家长中心，但规则完整度足以支撑当前需求。

## 4. 数据模型设计

### 4.1 `user_accounts`

继续作为统一登录账号表。

对家长角色采用以下约束：

1. `role = PARENT`
2. `username = 规范化手机号`
3. 初始密码为手机号后 6 位的加密结果
4. `status` 继续沿用现有 `ACTIVE` 机制，本阶段不额外开放启用/禁用操作

这样可以复用当前统一认证链路，不需要为家长端单独新建登录体系。

### 4.2 `parent_accounts`

继续作为家长档案表，表示“登录账号之外的家长业务身份信息”。

建议约束如下：

1. `phone` 作为家长主身份字段，必须唯一。
2. `user_account_id` 与家长登录账号一对一关联。
3. `display_name` 用于后台展示，默认优先取最近一次可用的监护人姓名；若缺失，则回退为系统占位名。

本阶段不额外拆分多联系方式、关系称谓、审核状态等更重的档案信息。

### 4.3 `parent_student_relations`

继续表示家长与学生的多对多关系，但新增绑定语义字段：

1. `binding_type = AUTO`
2. `binding_type = MANUAL`
3. `binding_type = AUTO_MANUAL`

同时增加唯一约束，保证同一对家长和学生只保留一条关系：

1. 唯一键：`(parent_account_id, student_id)`

三种绑定语义分别表示：

1. `AUTO`：由学员 `guardianPhone` 自动归并产生，可随手机号变化自动迁移或删除。
2. `MANUAL`：由后台人工维护，独立于 `guardianPhone` 自动规则。
3. `AUTO_MANUAL`：同时满足自动绑定和人工绑定；当自动条件失效时，只降级为 `MANUAL`，不直接删除。

### 4.4 数据语义收益

通过显式绑定类型，系统可以准确处理以下冲突场景：

1. 学员手机号变更时，只清理自动部分，不误删人工绑定。
2. 后台手动绑定一个本已自动绑定的学生时，可以升级为 `AUTO_MANUAL`。
3. 后台手动解绑 `AUTO_MANUAL` 时，只移除人工属性并降级为 `AUTO`。

## 5. 自动同步规则设计

### 5.1 同步触发点

以下场景都需要触发家长同步：

1. 新建学员成功后
2. 编辑学员成功后
3. Excel 导入学员成功后
4. 需要兼容老数据时的初始化迁移任务

同步入口建议由后端服务统一收敛，避免控制器、导入逻辑和业务页面各写一套规则。

### 5.2 规范化规则

同步前统一对手机号做规范化处理：

1. 去掉首尾空格
2. 去掉中间空格
3. 保留系统允许的手机号字符格式
4. 后端统一使用规范化结果作为家长身份匹配依据

这样可以避免因为格式差异造成同一手机号生成多个家长账号。

### 5.3 学员存在 `guardianPhone` 时

当学员档案存在有效 `guardianPhone` 时，同步逻辑如下：

1. 按手机号查找对应 `parent_accounts`
2. 若不存在，则自动创建：
   - `user_accounts` 家长登录账号
   - `parent_accounts` 家长档案
3. 登录账号用户名直接使用手机号
4. 初始密码为手机号后 6 位
5. 建立该家长与当前学员的绑定关系：
   - 若关系不存在，创建为 `AUTO`
   - 若原关系为 `MANUAL`，升级为 `AUTO_MANUAL`
   - 若原关系为 `AUTO` 或 `AUTO_MANUAL`，保持即可

### 5.4 学员 `guardianPhone` 变更时

当学员 `guardianPhone` 从旧手机号改为新手机号时，同步逻辑如下：

1. 识别旧手机号对应的家长档案
2. 处理旧家长与当前学员的关系：
   - 若为 `AUTO`，删除该关系
   - 若为 `AUTO_MANUAL`，降级为 `MANUAL`
   - 若为 `MANUAL`，不变
3. 再按新手机号执行一次标准自动绑定逻辑

这条规则直接满足“自动解除旧手机号家长与该学生的自动绑定，并自动绑定到新手机号家长”的确认结论。

### 5.5 学员清空 `guardianPhone` 时

当学员清空监护人手机号时：

1. 删除该学员相关的纯自动绑定 `AUTO`
2. 将 `AUTO_MANUAL` 降级为 `MANUAL`
3. 保留所有纯人工绑定 `MANUAL`

### 5.6 后台人工绑定时

后台人工绑定学生到家长时：

1. 若该关系不存在，创建为 `MANUAL`
2. 若该关系为 `AUTO`，升级为 `AUTO_MANUAL`
3. 若该关系已为 `MANUAL` 或 `AUTO_MANUAL`，直接提示已存在，不重复创建

### 5.7 后台人工解绑时

后台人工解绑学生与家长时：

1. 若关系为 `MANUAL`，直接删除
2. 若关系为 `AUTO_MANUAL`，降级为 `AUTO`
3. 若关系为 `AUTO`，不允许后台直接解绑，需提示“该关系由学员监护人手机号自动生成，请修改学员档案中的监护人手机号后再解除”

该规则可以避免后台误以为解绑成功，但系统又立即依据 `guardianPhone` 自动补回。

## 6. 后台页面设计

### 6.1 菜单与路由

管理后台左侧菜单新增“家长管理”，与以下模块同级：

1. 学员管理
2. 教练管理
3. 课程管理
4. 考勤管理
5. 训练记录
6. 阶段评估

前端新增独立路由和页面，沿用当前后台列表页的实现风格。

### 6.2 家长列表页

列表页建议展示以下字段：

1. 家长 ID
2. 展示名称
3. 手机号
4. 登录账号
5. 已绑定学生数
6. 绑定学生摘要
7. 最近登录时间
8. 更新时间

### 6.3 列表筛选

首版建议支持以下筛选项：

1. 按展示名称搜索
2. 按手机号搜索
3. 按绑定学生姓名搜索

如需进一步增强，可在二期追加“是否有绑定学生”等筛选项，但本阶段不是必须项。

### 6.4 列表操作

首版建议提供以下操作：

1. 详情
2. 绑定学生
3. 解绑学生
4. 修改密码
5. 重置密码

本阶段不提供删除账号和启用/禁用操作。

### 6.5 家长详情

详情页或详情弹窗建议分为两块：

1. 家长档案信息
   - 展示名称
   - 手机号
   - 登录账号
   - 最近登录时间
   - 更新时间
2. 绑定学生列表
   - 学员姓名
   - 学号
   - 当前监护人手机号
   - 绑定类型

绑定类型需以可读标签形式展示：

1. 自动绑定
2. 人工绑定
3. 自动 + 人工

### 6.6 绑定交互

“绑定学生”建议采用弹窗式交互：

1. 支持按学员姓名或学号搜索
2. 选中学生后创建人工绑定
3. 若原关系为 `AUTO`，则升级为 `AUTO_MANUAL`
4. 前端应明确提示绑定结果，避免后台误解为重复建关系

“解绑学生”建议区分不同绑定类型的反馈：

1. `MANUAL`：直接解绑成功
2. `AUTO_MANUAL`：提示“已移除人工绑定，自动绑定仍保留”
3. `AUTO`：按钮禁用或弹出说明，不允许直接解绑

## 7. 后端接口设计

### 7.1 家长管理接口

建议新增一组后台专用家长管理接口，与家长小程序门户接口分开：

1. `GET /api/v1/parents`
   - 分页查询家长列表
   - 支持展示名、手机号、学生姓名等筛选
2. `GET /api/v1/parents/{id}`
   - 查询家长详情
   - 返回基础档案与绑定学生明细
3. `POST /api/v1/parents/manual-bindings`
   - 新增人工绑定
4. `DELETE /api/v1/parents/{id}/students/{studentId}/manual-binding`
   - 解除人工绑定

### 7.2 同步接口或内部服务入口

建议增加统一同步服务入口，供以下流程内部调用：

1. 学员新增
2. 学员编辑
3. 学员导入
4. 数据初始化迁移

若需要显式接口，也可提供：

1. `POST /api/v1/parents/sync-by-student/{studentId}`

但该能力更适合作为内部服务方法，而非后台常规页面操作入口。

### 7.3 密码管理接口

家长密码管理继续复用现有账号接口：

1. `PUT /api/v1/user-accounts/password`
2. `POST /api/v1/user-accounts/reset-password`

需要补充的仅是后台“家长管理”页面对家长用户名为手机号的适配。

## 8. 小程序同步设计

### 8.1 登录方式

小程序家长端继续走统一登录页与统一认证接口，不新增单独登录入口。

需要同步调整的内容：

1. 家长登录账号认知统一为“手机号”
2. 登录文案明确提示“家长请使用手机号登录”
3. 初始密码提示为“手机号后 6 位”
4. 家长登录成功后仍按现有逻辑跳转至 `/pages/parent/home`

### 8.2 家长业务页面范围

本次不重做家长首页、预约、签到、成长档案等业务页面。

本次同步范围仅包括：

1. 登录身份统一
2. 账号来源稳定
3. 家长可见学生范围稳定
4. 多孩子切换继续可用

### 8.3 兼容策略

为平滑上线，需要兼容现有历史账号与历史关系：

1. 若存在 `parent_<phone>` 形式的旧用户名，需要迁移为纯手机号用户名
2. 若旧家长档案缺失登录账号，需要按手机号补建 `user_accounts`
3. 若旧绑定关系已存在但没有绑定类型，统一初始化为 `AUTO`

### 8.4 现有隐式登录兜底逻辑

当前 `ParentPortalService` 中存在“登录后按手机号自动兜底绑定学生”的逻辑。

本次设计要求将其收敛为“以显式同步结果为准”，原因如下：

1. 避免登录动作带来不可见的副作用
2. 避免每次登录重新补建或重复修正关系
3. 让绑定关系维护统一由学员同步逻辑和后台人工操作负责

## 9. 数据迁移与初始化

### 9.1 表结构调整

建议执行以下结构调整：

1. `parent_student_relations` 新增 `binding_type`
2. `parent_student_relations` 增加 `(parent_account_id, student_id)` 唯一约束
3. `parent_accounts.phone` 增加唯一约束

### 9.2 老数据迁移

建议补一轮初始化任务，完成以下工作：

1. 扫描已有 `parent_accounts`
2. 按手机号补建缺失的 `user_accounts`
3. 将 `parent_<phone>` 形式家长用户名迁移为纯手机号
4. 将历史关系默认标记为 `AUTO`
5. 按当前学员 `guardianPhone` 重算一次自动绑定，确保历史数据进入新规则体系

### 9.3 幂等性要求

迁移与同步逻辑都需要保证幂等：

1. 重复执行不会产生重复家长账号
2. 重复执行不会产生重复绑定关系
3. 重复执行只会修正状态，不会破坏人工绑定

## 10. 异常处理与边界策略

### 10.1 家长手机号存在但未绑定学生

允许登录成功，但家长首页需显示空状态，引导用户联系后台或等待绑定。

### 10.2 一个手机号绑定多个学生

允许登录成功，小程序继续显示多孩子切换。

### 10.3 一个学生绑定多个家长

允许多个手机号账号同时看到该学生，但账号和密码独立维护。

### 10.4 后台尝试解绑纯自动绑定

后端需返回明确提示，前端也应做显式说明，避免错误预期。

### 10.5 家长账号重复归并

后端必须保证同一手机号最终只对应一个家长账号与一个家长档案，禁止生成重复身份。

## 11. 测试与验收标准

### 11.1 自动建号与自动绑定

1. 新建学员并填写 `guardianPhone` 后，系统自动创建家长账号与家长档案
2. 自动创建的家长账号用户名为手机号
3. 初始密码为手机号后 6 位
4. 自动创建后该家长可登录小程序并看到对应学生

### 11.2 多对多关系

1. 同一手机号绑定两个学生时，家长登录后可看到两个孩子
2. 两个不同手机号都可绑定到同一学生
3. 不同家长登录后都能看到该学生，但账号互不影响

### 11.3 `guardianPhone` 变更

1. 学员监护人手机号从 A 改为 B 时，A 的纯自动绑定被移除
2. 学员监护人手机号从 A 改为 B 时，B 自动获得绑定
3. 若 A 对该学生还存在人工绑定，则变更后 A 仍保留可见权限

### 11.4 人工绑定与解绑

1. 后台手动绑定一个原本已自动绑定的学生时，关系升级为 `AUTO_MANUAL`
2. 后台解绑 `AUTO_MANUAL` 时，只去掉人工属性并降级为 `AUTO`
3. 后台解绑 `MANUAL` 时，关系被彻底移除
4. 后台不得直接解除纯 `AUTO` 绑定

### 11.5 密码与登录

1. 家长可用手机号 + 正确密码登录小程序
2. 后台修改密码后，新密码立即生效
3. 后台重置密码后，家长可使用新初始密码登录

### 11.6 历史数据兼容

1. 旧 `parent_<phone>` 账号迁移后可继续登录
2. 历史绑定不会因迁移被重复创建
3. 人工绑定和自动绑定在迁移后仍能按新规则运行

## 12. 后续计划衔接建议

本设计确认后，后续实施应优先拆解为以下几个工作块：

1. 数据结构与迁移脚本
2. 后端家长同步服务与后台管理接口
3. 管理后台家长管理页面
4. 小程序登录文案与兼容调整
5. 联调与回归验证

该拆分方式既能保证底层规则先稳定，又能让后台和小程序在同一套身份语义上完成同步。
